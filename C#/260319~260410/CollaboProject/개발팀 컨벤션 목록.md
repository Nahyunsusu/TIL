# 개발팀 컨벤션 목록

## 브랜치 전략

```
main      → 최종 결과물 (배포용)
develop   → 개발 통합

develop 브랜치의 하위로 개인 브랜치(개인 이름의 이니셜을 제목으로)를 생성하여 작업한다.
```
## 커밋 메시지 규칙
```
타입 : 작업 내용
```

### 타입 목록

- `feat` : 기능 추가
- `fix` : 버그 수정
- `refactor` : 구조 개선
- `docs` : 문서 수정
- `chore` : 기타 작업

### 예시

```
feat: 플레이어 이동 구현
fix: 총알이 벽을 통과하는 문제 수정
refactor: EnemyAttack 구조 개선
docs: README 업데이트
```

### 금지~~

```
수정
ㅇㅇ
fix
test
```

## PR 및 Merge 규칙
- 작업 완료 후 `develop`으로 병합
- 다른 사람 코드 위에 덮어쓰기 금지
- 충돌 발생 시 단독 해결 ❌ → 반드시 공유 후 해결
- 머지 할 시 머지 리퀘스트하고 팀장에게 알려주면 머지.

   ### 권장
    - 작은 단위로 자주 머지
    - 대규모 변경 한 번에 머지 ❌

### 기본 사이클
```
1. develop pull
2. feature 브랜치 생성
3. 작업 진행
4. commit
5. develop 병합
```

## 씬 관리 방식
- 씬은 한 씬당 한 사람이 담당해서 작업.
  
- 다른사람의 씬을 사용할 일이 있다면 담당자에게 알리고 사용

## 프리팹 관리 방식
- Player / Enemy / UI / Map 분리 작업
   - 프리팹은 각 항목의 분류에 맞는 곳에 만든다.

   👉 씬 충돌 최소화
   
- 프리팹을 원본을 Import된 원본을 가져가서 쓴다. 
  

## 작업 관리 도구 및 태스크 기록 방식

- 제공된 Notion과 Google Sheets 등을 이용하여 작업을 기록한다.
- 개인이 진행하고 있는 과정에 진행도를 체크하여 팀원들이 알 수 있게 한다.
